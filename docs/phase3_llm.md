## 1. 문제 (Problem)
- CNN은 라벨만 뱉음("잎곰팡이병 87%"), LSTM은 숫자만 뱉음("내일 내부온도 32℃") → 초보자는 **원인·조치**를 모름.
- 게다가 작물은 생물이라 **내 뜻대로 안 됨** → 초보자는 *"이게 병충해가 맞긴 한 건지"* 부터 헷갈림.
- 최초 기획의 두 기둥 = **① 초보자도 쉽게 키우기 + ② 병충해 탐지.** Phase 1·2 예측·진단을 받아 이 둘을 **사람 말로** 완성한다.
- **분업 원칙(불변):** *진단은 ML/DL, 설명·처방·코칭은 LLM.* LLM에게 진단을 시키지 않는다(환각 위험).

## 2. 컨셉 — 재배 도우미 loop ⭐
DL 진단을 *반응형 진단기*가 아니라 **"확인 단추"** 로 재정의 → 예방·확인·처방·상담이 한 흐름이 된다.
```
[평상시]  A. 일일 코치 ──── 매일 "오늘 할 일" (LSTM 환경 예측)
              │
[이상 징후] B. 조기 경보 ──── 환경 × 질병위험 → "지금 잎 확인해봐요"
              │               (또는 사용자가 직접 "어 이거 좀 이상한데?")
              ▼
[확인]    🔬 DL 진단 ──────── 잎 사진 → 병충해 맞는지 확인 (CNN·게이트·YOLO)
              │
              ├─ 정상 → C. 안심 "괜찮아요, 물자국이에요"
              └─ 질병 → C. 처방 + 용어풀이 + 📖 농사로 RAG 근거
              │
[후속]    D. 대화형 Q&A ───── "약 없이 환경으로만 잡을 수 있어요?"
```
> **DL = 확인 단추 / LLM = 그 앞(예방 코치·경보)과 뒤(처방·안심·상담)를 채우는 도우미.**
> 평소엔 챙겨주고, 이상하면 사진으로 확인받고, 결과를 사람 말로 설명+상담 → "쉽게 키우기 + 병충해 탐지" 완성.

## 3. 입력 — Phase 1·2가 넘겨주는 것 (Inputs)
실제 산출물 기준 (Phase 2에서 이미 만들어진 출력):

| DL/ML 출력 | 형태 | 예시값 | 모델·근거 |
|---|---|---|---|
| 🌡️ 환경 예측 | 다음날 내부온도·추세 | `32.4℃`, 야간 고습 | LSTM (MAE 1.18℃ < baseline) |
| 🔬 잎 진단 | 라벨 + 확률 | `잎곰팡이병 0.87` | `tomato_resnet18.pt` (4분류 acc 0.96 · ROC-AUC 0.997) |
| 🚪 입력 게이트 | OOD / 부위 판정 | "잎 아님" 차단 · "과실(0.9)" 차단 | OOD 게이트 + `tomato_part.pt` |
| 📍 병변 위치 | 박스 리스트 | 잎 5장 중 3장 병변·아래쪽 집중 | `tomato_yolov8n.pt` (mAP@50 0.78) |
| 🔥 근거 시각화 | Grad-CAM 히트맵 | 모델이 본 부위 오버레이 | resnet18 `layer4` |
| 📖 재배 지식 | RAG 검색 결과 | 농사로 방제 가이드 chunk | 농사로 코퍼스 4종 (`data/nongsaro/*.md`) |

## 4. 기능 (Features)
| # | 기능 | 기둥 | 쓰는 입력 |
|---|---|---|---|
| **A** | **일일 재배 코치** — 매일 "오늘 할 일" 일상 코칭 | ① 쉽게 키우기 | LSTM 환경 예측 |
| **B** | **병충해 조기 경보** — 환경×질병위험 → 선제 점검 유도 | ①+② 다리 | LSTM + RAG 도메인지식 |
| **확인** | **DL 진단(확인 단추)** — 잎 사진 → 병충해 맞는지 확인 | ② 병충해 | CNN·게이트·YOLO·Grad-CAM |
| **C** | **초보자 눈높이 언어** — 용어풀이 + 정상 시 안심 | ① 쉽게 키우기 | 진단 결과 + 프롬프트 |
| **D** | **대화형 Q&A** — 일반 재배 상담 | ① 쉽게 키우기 | 농사로 RAG |

> 스마트팜은 센서 기반이라 환경 스트림이 들어온다는 전제 → A·B 성립. 데모는 환경 시나리오 입력, 실배포는 센서 연동(→ PRD §10).

## 5. 설계 (Method) — 확정 방향

### 5-1. DL 출력 → LLM 입력 매핑 (Function Calling)
ML/DL 결과를 자연어로 풀어 넣지 않고 **구조화 함수 입력**으로 안정 전달:
```
get_forecast()   → { next_temp, trend, humidity_risk }     # A·B
get_diagnosis()  → { label, prob, probs{4종}, part, ood_blocked }  # 확인
get_detection()  → { boxes:[{cls, conf, region}], lesion_count }   # 확인
```

### 5-2. RAG (검색 증강) — 농사로, **잎병 3종 범위**
- 코퍼스: 농사로 재배·방제 가이드 중 **잎곰팡이병·황화잎말이·잎마름역병(+토마토 일반 재배)** 4문서. 잎마름역병은 진단 4분류 확장(PR #3)과 함께 추가.
- 흐름: 진단 라벨/질문 → 관련 가이드 chunk 검색(임베딩·코사인 유사도) → 근거로 프롬프트 결합 → **처방에 출처 인용**.

### 5-3. 환각 방어 3종 ★ (이 단계의 신뢰성 포인트)
DL이 주는 *확률·게이트·클래스 목록*을 그대로 활용해 "AI가 모를 땐 모른다고 말하게":
1. **신뢰도 톤 분기** — `prob` 구간별 강도 조절. 고확신(>0.8)=단정·즉시 방제 / 애매(<0.6)=단정 금지·정밀 확인 권유.
2. **게이트 차단 사유 안내** — 차단 시 그냥 막지 않고 *왜·어떻게 다시 찍을지* 안내("과실 사진이라 잎병 진단 불가 → 잎 뒷면을 밝은 곳에서").
3. **클래스 한정성 자기인지** — 모델이 아는 범위(잎병 3종+정상, 4종뿐)를 system 프롬프트에 고정. 범위 밖 질문엔 "진단 보류"라고 정직하게.

### 5-4. 시간축 처방 (진단 × LSTM)
진단 단독이 아니라 **환경 예측과 교차**해 선제 처방.
예: 잎곰팡이병(고습 악화) + LSTM 야간 고습 예측 → "오늘 밤 환기 예약, 새벽 결로 위험". → ML+DL+LLM 세 단계가 한 문장에서 만남.

### 5-5. 출력 형식 (구조화 JSON → 화면·알림 공유)
처방을 고정 스키마로 받아 Streamlit 카드와 디스코드 메시지가 같은 소스 사용:
```
{ 진단요약, 원인, 즉시조치, 예방, 재촬영시점, 근거출처[] }
```

### 5-6. 통합 파이프라인
```
센서 스트림 + 잎 사진
  → Phase1 ML(작물/환경) + Phase2 DL(게이트→진단·YOLO·Grad-CAM·LSTM 예측)
  → Phase3 LLM(function calling 입력 + 농사로 RAG 근거 + 환각 방어)
  → 자연어 처방·코칭(구조화)
  → Streamlit 대시보드 + 디스코드 알림
```

## 7. 범위·한계 (Scope) — 정직성
- **진단 = 잎병 3종(+정상).** 출발은 AI Hub 071의 잎병 2종(잎곰팡이병·황화잎말이) — 071 질병이 area=잎에만 존재해 과실·꽃·줄기 병해는 제외. 이후 **잎마름역병(late_blight)을 PlantVillage(CC0) 혼합으로 확장**해 4분류가 됨(PR #3) — "데이터 추가 시 클래스+코퍼스 동반 확장" 경로를 실제로 밟은 사례.
- 이는 **의도된 범위** — 게이트(5-3-②③)로 범위 밖 입력을 정직하게 차단/보류.

## 8. 완료 기준
- 평상시 코치(A)·조기경보(B)가 돌고, 사진 확인(DL)→처방·안심(C)→상담(D)까지 한 흐름.
- 처방에 **근거 출처**(농사로) 동반, 저확신·범위밖 케이스에서 단정하지 않음.
- `app/phase3_llm.py` 대시보드 + 디스코드 알림 동작.

## 9. 결과 (Result)
- **처방(C)**: 잎 사진 → 게이트→진단→function calling→RAG 근거→구조화 JSON 처방. 근거출처는 LLM이 아니라 **코드가 채움**(환각 배제), 스키마 위반 시 1회 재시도 후 안전 폴백.
- **일일 코치(A)·조기 경보(B)**: `src/llm/pipeline.py` — LSTM 환경 예측 기반, 습도위험 시 RAG 근거를 경보에 주입.
- **시간축 처방(5-4)**: 습도 민감 병해(잎곰팡이병·잎마름역병) 진단 시 환경 예측을 교차 주입 — ML+DL+LLM이 한 처방에서 만남.
- **자동 감시**: `src/llm/monitor.py` — 가상센서 스트림 재생 → 임계 판정 → 디스코드 Webhook 알림(레벨 전이 시만 발송, 전송 실패 시 재시도).

## 10. 배운점 (Learned)
- **환각은 프롬프트가 아니라 구조로 막는다** — 근거출처를 LLM 출력에서 받지 않고 RAG 검색 결과로 코드가 채우는 것, `format=` JSON 스키마 강제 + 검증 재시도가 프롬프트 지시보다 확실했다.
- **function calling은 "자연어로 풀어 넣기"의 상위호환** — DL 결과를 구조화 입력으로 넘기니 tool 실패까지 모델에 되먹여 복구 가능.
- **RAG는 스코프가 품질** — disease 라벨로 검색 범위를 한정하고, 매칭 없으면 빈 결과를 반환(엉뚱한 병해 근거 주입 방지)하는 게 top-k 점수보다 중요했다.
- **도메인 갭 실전 경험** — 예시 사진만 PlantVillage로 바꾸자 정상 잎을 오진 → 모델도 PV 혼합 재학습으로 해결. "데모 입력 분포 = 학습 분포" 원칙 체득.
- **알림은 dedup이 본체** — 매 틱 발송이 아니라 레벨 전이 시만 발송 + 실패 시 재시도(유실 방지)가 실사용 가능한 알림의 조건.
